RAG 学习

RAG 解决的核心问题

通用大语言模型（LLM）存在四大痛点：

  1. 知识过时：训练数据有截止时间，无法自动更新实时信息。
  2. 领域知识匮乏：训练数据多来自公开网络，无法覆盖企业内部/专业领域知识。
  3. 幻觉问题：模型会生成看似合理但实际错误的内容。
  4. 数据安全：直接将敏感数据输入模型存在泄露风险。

RAG（Retrieval Augmented Generation，检索增强生成） 正是为解决这些问题而诞生的技术。

RAG 核心定义

  • 本质：在生成回答前，先从外部知识库检索相关信息，用这些信息修正和补充模型的回答。
  • 公式：RAG = 检索技术 + LLM 提示
  • 核心思想：不修改模型本身，而是通过优化输入的提示词，将检索到的事实资料作为上下文喂给模型，让回答更准确、更贴合业务。

RAG 工作原理（核心流程）

1. 离线准备线（知识库预处理）

负责将文档变成可检索的向量数据：

  1. 文档加载：加载各类格式文件（TXT、PDF、Word等）。
  2. 文本分割：将长文档切分成固定长度的文本块（chunk）。
  3. 向量化：将每个文本块转换为向量（embedding）。
  4. 存储：将向量存入向量数据库（如 ChromaDB），建立索引。

2. 在线服务线（用户提问处理）

响应用户的实时查询：

  1. 用户提问：接收用户问题。
  2. 检索环节：将用户问题向量化，在向量数据库中匹配最相似的文本块。
  3. Prompt 融合：将检索到的文本块和原始问题组合成新的提示词。
  4. 大模型回答：将组合后的提示词输入 LLM，生成最终答案。

RAG 标准三阶段

1. 索引阶段（Indexing）

  • 目标：构建可检索的知识库。
  • 步骤：
    • 加载文件 → 提取文本内容 → 分割为 chunk → 文本向量化 → 存入向量数据库。

2. 检索阶段（Retriever）

  • 目标：找到与用户问题最相关的知识片段。
  • 步骤：
    • 将用户 query 向量化 → 在向量库中做相似度匹配 → 召回 top-k 个最相关的 chunk。

3. 生成阶段（Generation）

  • 目标：基于检索结果生成准确回答。
  • 步骤：
    • 将检索到的 chunk 作为上下文，与用户问题一起拼接成 Prompt → 输入 LLM → 生成最终回答。

RAG 核心价值

  1. 解决知识时效性：接入最新文档（财报、政策），让模型输出“与时俱进”的信息。
  2. 降低模型幻觉：回答基于检索到的事实资料，大幅减少编造内容的概率。
  3. 低成本高效：无需重新训练模型（Fine-tuning），只需更新知识库，成本更低、迭代更快。
  4. 增强数据安全：敏感数据保存在内部知识库，仅检索相关片段，避免直接泄露。

总结

  • RAG 是提示词工程的进阶应用：通过检索外部知识，让提示词包含更精准、更丰富的上下文。
  • RAG 是 LangChain 框架的核心应用场景：LangChain 提供了文档加载、分割、向量库、检索、Prompt 模板等全链路组件，是实现 RAG 的首选工具。
