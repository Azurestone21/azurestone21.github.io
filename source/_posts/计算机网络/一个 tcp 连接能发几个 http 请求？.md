理论上，一个TCP连接可以发送无限个HTTP请求。

实际上，一个 TCP 连接能发送的 HTTP 请求数量，核心取决于 HTTP 版本。

## 1. HTTP/1.0：“一发一断”

HTTP/1.0 在默认情况下，一个TCP连接只发一个HTTP请求。每个请求响应结束后就会断开TCP连接。效率非常低。

- **如何工作：**
  1. 与服务器建立一个TCP连接
  2. 通过这个连接发送一个HTTP请求
  3. 接收服务器的响应
  4. **立即关闭这个TCP连接**
- **缺点**：
  - 建立一个TCP连接需要“三次握手”，成本很高。如果一个网页有10个图片，就需要建立和断开10次TCP连接，非常浪费资源和时间。

为了解决这个问题，一些浏览器引入了非标准的 `Connection: keep-alive` 头，试图重用连接，但这并非官方标准，且受服务器端超时配置（如 KeepAliveTimeout）限制，并非无限复用。

## 2. HTTP/1.1：“持久连接” (Keep-Alive)

HTTP/1.1 将持久连接`Connection: keep-alive`设为**默认行为**。这意味着TCP连接在完成一次请求-响应后不会立即关闭，而是保持打开状态，允许客户端继续发送新的请求。

- **如何工作**： 客户端发送请求，服务器回复响应，然后**通道保持空闲**，等待下一个请求。客户端会按顺序一个接一个地发送请求。
- **如何关闭**： 服务器或客户端可以通过发送 `Connection: close` 头来主动关闭连接。否则，连接可能会在一段空闲超时后被关闭。
- **缺点：**
  - **队头阻塞（Head-of-Line Blocking）**： 单个 TCP 连接可被复用，发送 多个 HTTP 请求，但请求需串行执行（必须等前一个请求的响应返回后，才能发送下一个请求）。如果第一个请求处理得很慢（比如一个大文件），它就会“阻塞”后面已经准备好的请求。后面的请求必须排队等待。
  - **连接的复用并非无限**：服务器端通常会配置 “最大请求数”（如 Nginx 的 keepalive_requests，默认 100 个）和 “连接超时时间”（如 keepalive_timeout，默认 65 秒）—— 若单个连接发送的请求数达到上限，或连接空闲时间超过超时阈值，TCP 连接会被关闭；

## 3. HTTP/2：“多路复用” (Multiplexing)

HTTP/2是革命性的升级。它允许在**一个TCP连接上同时交错地发送多个请求和响应**，而不是串行。

- **如何工作**： HTTP/2将请求和响应消息分割成更小的**帧（Frames）**，并给每个帧打上一个**流（Stream）** 的ID编号。所有这些来自不同流的帧都可以在一个TCP连接上混合在一起传输。接收方根据流ID就能重新组装出完整的请求和响应。
- **优点**：
  - **彻底解决了HTTP层面的队头阻塞问题**。慢请求的帧不会阻塞其他请求的帧。
  - 只需要一个TCP连接就能加载整个页面，减少了TCP握手带来的延迟和开销。
  - 更高效地利用TCP带宽。
- **限制**： 虽然HTTP层面没有队头阻塞了，但**TCP层面依然存在**。因为TCP保证数据包必须按顺序送达。如果任何一个TCP数据包在传输中丢失，整个TCP连接必须等待该包重传成功，这会阻塞所有正在传输的HTTP/2流。

## 4. HTTP/3：“基于QUIC”

为了彻底解决TCP层面的队头阻塞问题，HTTP/3迈出了更大的一步：它弃用了TCP，转而使用基于UDP的新协议——**QUIC**。

- **如何工作**： QUIC在UDP之上实现了自己可靠的、有序的传输机制。它在每个“流”上保证数据的顺序，但如果一个流丢失了数据包，只会影响该流本身，而不会影响连接上的其他所有流。
- **优点**：
  - **解决了TCP和HTTP两个层面的队头阻塞问题**。
  - 连接建立更快（通常0-RTT或1-RTT），尤其在高延迟网络下优势明显。

| **HTTP 版本** | **底层连接**        | **单个连接支持的请求数**           | **核心限制因素**                            | **关键特性**              | **主要问题**           |
| :------------ | ------------------- | :--------------------------------- | :------------------------------------------ | :------------------------ | :--------------------- |
| HTTP/1.0      | TCP                 | 1 个（默认）/ 有限个（keep-alive） | 无默认长连接，超时配置                      | 短连接，无复用            | 连接无法复用，开销巨大 |
| HTTP/1.1      | TCP                 | 多个（串行）                       | 服务器 `keepalive_requests`<br/> 配置、超时 | 长连接，串行复用          | HTTP队头阻塞           |
| HTTP/2        | TCP                 | 海量（并行）                       | 服务器最大并发流数、资源                    | 二进制帧，多路复用        | TCP队头阻塞            |
| HTTP/3        | **QUIC (over UDP)** | 海量（并行）                       | 服务器资源、QUIC 配置                       | QUIC 多路复用，无队头阻塞 | 暂无明显通用性问题     |
