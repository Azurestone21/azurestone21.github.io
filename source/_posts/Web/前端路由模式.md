在前端开发中，路由模式主要有两种：**Hash 模式**和 **History 模式**，它们用于实现单页应用（SPA）中不同视图之间的无刷新切换。

## 一、Hash模式

**原理**

利用`window`的`location.hash`属性和`hashchange`事件，可以实现监听浏览器地址`hash`值（URL 中 # 及其后面的部分）变化，执行相应的 js 切换网页内容。

**特点**

- URL 中包含 #，例如 http://example.com/#/home。
- 哈希值不会随请求发送到服务器端，所以改变哈希值，不会重新加载页面。
- 兼容性好，支持所有浏览器（包括低版本 IE）。
- 部分搜索引擎可能会忽略 # 后面的内容，影响页面的搜索引擎优化。

在 `html5` 的 `history` 出现前，基本都是使用 `hash` 来实现前端路由的。

## 二、History模式

**原理**

使用 HTML5 History API 的`pushState()`、`replaceState()`**在不刷新页面的情况下修改浏览器地址栏 URL，并将当前状态（如路径、自定义数据）存入 “历史记录栈”。**并且通过`popstate`事件监听浏览器历史记录栈的变化，获知浏览器前进 / 后退操作。

注意：`pushState()`、`replaceState()` 仅修改历史记录栈内容或指针，不会主动触发任何页面更新，它们本身不会触发 popstate。调用 history.back()、history.forward()、history.go() 方法时，才会触发 popstate 事件。

**特点**

- URL 更美观，没有 #，例如 http://example.com/home。
- 需要服务器配合，否则刷新页面时可能会出现 404 错误。
- 兼容性稍差，不支持 IE9 及以下版本。
- URL 结构更符合搜索引擎的抓取规则，对 SEO 更友好。

### 为什么history模式会出现 404 的情况

history 模式开发的SPA项目，需要服务器端做额外的配置，否则会出现刷新白屏。

**原因**

history模式没有`#`，刷新页面时，URL 包括页面路由都会随请求发送给服务器，但是单页面应用只有一个 `index.html`，路由对应的文件资源不存在，就会报 404。

**解决**

由后端做一个保底映射，让服务器将所有路由请求转发到`index.html`。

## 总结

Hash 模式：适合兼容性要求高、不需要美观 URL、快速开发的场景（如内部系统）。

History 模式：适合追求 URL 规范性、注重 SEO、能配合服务器配置的场景（如面向用户的公开网站）。
